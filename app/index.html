What it is · how it works

An agent that measures before it trusts

What is Glasswork? A thoughtful productivity agent for document work. Instead of betting on one big model and re-reading everything it says, Glasswork treats model choice as a measurement problem: it grades every open model on your actual task, picks the cheapest one that clears a quality bar you set, and hands you a cryptographic receipt for every call. You get the cheapest reliable answer and a way to audit it.

The deterministic core

Deterministic where it counts

A custody receipt only means something if the computation around it is reproducible. Glasswork's scoring and custody are fully deterministic — given a recorded output, the same rubric score and the same Merkle root recompute identically, every ingest. The model step runs at temperature 0 (a best-effort knob, not a guarantee) — which is exactly why we hash and bind the bytes that actually ran instead of trusting the model to repeat itself.

A Fractal Custody Object binds bytes, not just text — so the same spine scales past a single inference. An entire paper becomes a Fractal Custody Graph over all its parts: text, figures, tables, datasets, images, audio, any media. Bind every part and you can check the whole artifact against a gold standard — not "looks right," but provably the same, byte for byte. This demo is one leaf of that graph; the method is the same all the way up.

The error surface — our edge

Don't just show a result. Show where it's wrong.

Most paper-to-result agents hand you an answer and stop. Glasswork hands you the answer plus its error surface: measured against a gold standard, exactly which facts each model recovered, which it missed, and a flag on any claim whose evidence isn't in the source (a would-be hallucination). Pick a model and see precisely where it fails.

Prove it

A custody receipt for every call — recompute it yourself

Each inference emits two hash leaves — an operational leaf (model, tokens, cost) and a content leaf (hashes of the exact prompt and output bytes) — chained into a per-model Merkle root with RFC 6962 domain separation (leaf 0x00, node 0x01; duplicate-last pairing). Press Verify and your browser re-derives the root from the named bytes and compares. Reject iff mismatch.

The AI stack

What's under the hood

Models — 12 open weights, 1B → 235B

Five families through Butterbase's OpenAI-compatible gateway, same prompt, temperature 0 for reproducibility.

Scoring — deterministic rubric

Precision penalizes any claim whose supporting span isn't in the document (hallucination); recall measures coverage against the gold set. No LLM judge in the loop.

Selection — cheapest that passes

The agent picks the lowest-cost model meeting the pre-set precision/recall bar on the corpus — a decision recorded as data.

Custody — Fractal Custody Objects

Two hash leaves per call → a per-model Merkle root, domain-separated (leaf 0x00 / node 0x01). Published method, cited; verifiable in-browser.

Backend — 100% Butterbase

AI gateway (inference), Postgres + Data API (catalog, gold set, 24 runs, 60 custody records), a serverless function (results), and the frontend deployment serving this page.

What we can't prove

Honest ceilings — printed, not buried

This honesty is the point. An agent you can trust is one that tells you the edges of what it measured.

Accuracy is on this task/corpus.

The numbers are extraction quality on these documents (N shown), not a universal capability claim about any model.

Custody proves provenance, not correctness.

A receipt tells you exactly which model produced which bytes at what cost. It does not say the answer is right, and it does not reproduce non-deterministic inference.

Cost is Butterbase-reported.

Savings are auditable computation from recorded token counts and the gateway's usage figures — trustworthy, but not an independently proven benchmark.

An LLM verdict is a signal, never a proof.

A model saying "this looks wrong" is a detector-arm signal. The custody spine tells you which bytes produced this. They are different things.

FAQ

Questions a judge should ask

Is this real data, or a mockup?

Real. Every number is 24 live inferences run through Butterbase's AI gateway and served from the app's own database — you can read it yourself at the public results function this page fetches.

Does a custody receipt mean the answer is correct?

No — and we say so on the page. Custody proves provenance: which model produced which exact bytes, at what token count and cost, tamper-evident. Whether the extraction is right is a separate question, measured by the precision/recall score.

Why not just use the biggest model?

Because bigger didn't win here. A 27B Gemma beat both the 70B and the 235B on recall, and the 70B Llama emitted invalid JSON on one document — unusable by an automated pipeline. You only find out by measuring on your task.

Is precision really 100%?

For every model that returned valid, parseable output, yes — none kept an ungrounded (hallucinated) span on this corpus. The 70B's score covers only the one document it could be parsed on; its recall (44%) honestly reflects the failure on the other.

What exactly runs on Butterbase?

All of it: inference (AI gateway), data (Postgres + auto-generated Data API), backend logic (a serverless function), and this website (frontend deployment). No other backend.

Can I verify the custody myself?

Yes — press Verify in the section above; your browser recomputes the Merkle root with SHA-256 and rejects a tampered one (try selecting openai/gpt-oss-20b). Or download the full dataset and re-derive every root offline.
